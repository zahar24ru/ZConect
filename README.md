# ZConect

Легковесная система удаленного доступа для Windows 10/11 через WebRTC (IPv4), с кодеком VP8 и сервером сигналинга на Ubuntu 24 LTS.

## Цели первой версии (MVP)

- Соединение двух ПК за NAT через WebRTC.
- Получение одноразовых кодов сессии: `логин(8 цифр)` + `пароль(8 цифр)`.
- Полный удаленный контроль: видео, клавиатура, мышь.
- Обязательная синхронизация буфера обмена.
- Простой обмен файлами.
- Настройки качества изображения.
- Настройки сетевых параметров (Server URL, STUN/TURN, IP/порт, relay режим).
- Запись логов в `logs.log`.

## Статус

Рабочий MVP уже есть:

- Сервер сигналинга: HTTP API (`create/join/close`) + WebSocket relay `/ws`.
- Клиент WPF (.NET 8): создание/подключение по кодам `8+8`, настройки сети/STUN/TURN/FFmpeg.
- WebRTC видео: `host` шарит экран, `viewer` получает поток (в отдельном окне `RemoteScreenWindow`).
- Удалённое управление: `viewer` отправляет mouse/keyboard по `dc-input`, `host` применяет через WinAPI.
- Авто-деплой сервера из Windows одной командой: `deploy/deploy_from_windows.ps1`.
- Рабочий двусторонний `dc-clipboard` (только текст, лимит 256 KB, авто-синхронизация).
- Рабочий двусторонний `dc-file` (MVP): отправка файла в обе стороны, прогресс в UI, авто-сохранение у получателя.
- В настройках добавлена вкладка `debug-log` с фильтрацией шумных DEBUG-событий.
- Клиент запускается с запросом прав администратора (UAC, `requireAdministrator`).
- В `debug-log` добавлен переключатель политики `PromptOnSecureDesktop` для видимости UAC-запросов в удаленной сессии.
- Добавлен авто-режим ICE для viewer: `LAN(3с) -> srflx(4с) -> relay(5с)` с fallback.
- Внизу главного окна показывается ICE-статус (`host/srflx/relay`) с IP и debug-список последних 5 ICE кандидатов.
- При штатном закрытии клиента сессия автоматически закрывается через `POST /api/v1/session/close`.
- В `Настройки -> Система` и в `RemoteScreenWindow` выбор `DisplayId` сделан через выпадающий список обнаруженных дисплеев (fallback `DISPLAY1`).
- Добавлена синхронизация типа курсора host -> viewer (arrow/ibeam/hand/resize/wait и т.д.) через `dc-control`.

Ещё не доделано (следующий этап): боевой `ws_token`, Redis (вместо in-memory), расширение `dc-file` (ack/error/cancel/resume).

## Требования

- **Клиент:** Windows 10/11, .NET 8.
- **Сервер:** Ubuntu 24 LTS (или аналог), Docker и Docker Compose; для NAT — STUN/TURN (coturn).
- **Опционально:** ffmpeg для VP8 Probe.

## Структура репозитория

- `client/` — модули Windows-клиента.
- `server/` — сервер сигналинга и сессий.
- `deploy/` — инфраструктура и автоматизация деплоя.
- `docs/` — единый источник проектных решений.

## Деплой сервера

Цель — максимально автоматизированный запуск одной командой (см. `docs/DEPLOY_UBUNTU24.md`).

## Принципы разработки

- Никаких "монолитных" файлов на сотни строк.
- Один модуль — одна ответственность.
- Все изменения архитектуры сразу фиксируются в `docs/`.
- Совместимость в MVP: только IPv4.
